# Spark 数据倾斜笔记

## 1. 核心概念

**数据倾斜** 是指在进行 shuffle 操作（如 join、groupBy、reduceByKey）时，数据分布不均匀，导致某些 task 处理的数据量远大于其他 task，从而拖慢整个作业的执行速度，甚至引发 OOM。

**根本原因**：数据本身分布不均，某个或某几个 key 对应的数据量远大于其他 key。

**典型现象**：
- 大部分 task 很快完成，少数 task 执行缓慢或失败
- 作业整体耗时远大于预期
- 出现 OOM 或 GC 频繁

## 2. 关键机制

### 2.1 数据倾斜的产生环节
- **shuffle 操作**：数据需要重新分区时，key 的哈希分布不均
- **join 操作**：大表与大表 join，或大表与小表但 key 分布不均
- **聚合操作**：groupBy、reduceByKey 等

### 2.2 大 key 定位方法
- 使用 `sample` 算子对 RDD 采样，统计每个 key 的数据量
- 在 Spark SQL 中通过 `GROUP BY key` 并排序，找出数据量异常的 key

## 3. 常见问题与优化

### 3.1 方案一：Hive ETL 预处理

**适用场景**：数据倾斜源于 Hive 表，且业务频繁使用 Spark 分析该表。

**思路**：在 Hive 阶段预先对数据按 key 聚合或 join，Spark 直接使用预处理后的中间表，避免 shuffle。

**优点**：实现简单，效果显著，可提升性能 6 倍以上。

**缺点**：治标不治本，Hive ETL 阶段仍可能发生数据倾斜。

### 3.2 方案二：过滤少数倾斜 key

**适用场景**：导致倾斜的 key 只有少数几个，且对计算结果影响不大。

**思路**：通过 `filter` 或 `WHERE` 子句直接过滤掉这些 key。

**优点**：实现简单，完全规避数据倾斜。

**缺点**：适用场景有限，多数情况下倾斜 key 较多。

### 3.3 方案三：提高 shuffle 并行度

**适用场景**：作为数据倾斜的初步尝试手段。

**思路**：
- RDD：`reduceByKey(1000)` 设置分区数
- Spark SQL：设置 `spark.sql.shuffle.partitions`（默认 200）

**原理**：增加 task 数量，让每个 task 处理更少的数据。

**优点**：实现简单，可缓解数据倾斜。

**缺点**：无法彻底解决极端倾斜（如单个 key 数据量极大）。

### 3.4 方案四：两阶段聚合（局部聚合 + 全局聚合）

**适用场景**：聚合类 shuffle 操作（reduceByKey、groupBy）。

**思路**：
1. 给每个 key 打上随机前缀（如 1_hello, 2_hello）
2. 进行第一次局部聚合
3. 去掉随机前缀，进行第二次全局聚合

**优点**：有效解决聚合场景下的数据倾斜。

**缺点**：仅适用于聚合操作，不适用于 join。

### 3.5 方案五：广播 Join

**适用场景**：大表 join 小表，或大表 join 大表中少数倾斜 key。

**思路**：
- 自动广播：设置 `spark.sql.autoBroadcastJoinThreshold`
- 手动广播：使用 `broadcast` 提示

**优点**：避免 shuffle，彻底解决 join 倾斜。

**缺点**：小表需能放入内存。

### 3.6 方案六：拆分大 key + 扩容小表

**适用场景**：大表 join 大表，且存在少数倾斜 key。

**思路**：
1. 将大表中的倾斜 key 拆分（加随机前缀）
2. 将小表中对应的 key 扩容（复制多份）
3. 分别 join 后 union 结果

**优点**：解决大表 join 大表的数据倾斜。

**缺点**：实现复杂，需预先识别倾斜 key。

## 4. 学习建议

1. **先定位，再优化**：使用 `sample` 或 SQL 分析找出倾斜 key，避免盲目优化。
2. **从简单方案入手**：优先尝试提高并行度或过滤倾斜 key。
3. **区分场景**：聚合场景用两阶段聚合，join 场景用广播 join 或拆分扩容。
4. **考虑上游预处理**：如果数据源可控，在 Hive 阶段预处理是最彻底的方案。
5. **结合执行计划分析**：使用 `EXPLAIN` 查看物理执行计划，确认 shuffle 操作和分区情况。
6. **关注资源调优**：数据倾斜常与资源不足叠加，合理设置内存、并行度、缓冲区大小。
