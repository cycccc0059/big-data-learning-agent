# 数据倾斜处理方案

## 1. 核心概念

**数据倾斜**是指在分布式计算过程中，数据被分配到各个节点时，部分节点分到的数据量远大于其他节点，导致整体计算效率被少数“慢节点”拖累的现象。

- **本质**：数据分布不均匀 → 计算负载不均衡 → 木桶效应（整体耗时取决于最慢节点）
- **常见场景**：Join、GroupBy、Count Distinct、Reduce 阶段
- **典型表现**：任务进度卡在 99%、部分节点 OOM、整体运行时间远高于预期

## 2. 关键机制

### 2.1 数据倾斜的产生原因

| 原因类型 | 具体场景 | 示例 |
|---------|---------|------|
| 键值分布不均 | 热点 Key 出现 | 某个城市用户量占 80% |
| 业务逻辑导致 | 空值/默认值过多 | user_id 为 null 的数据全部进入同一个分区 |
| 数据膨胀 | 小表与大表 Join 时，小表 Key 重复 | 维度表中某个 Key 对应大量事实数据 |
| 分区策略不当 | Hash 分区无法打散数据 | 按日期分区但某天数据量暴增 |

### 2.2 分布式计算中的影响链路

```
数据输入 → Shuffle（分区/排序） → Reduce 阶段
                ↓
        数据倾斜发生在 Shuffle 阶段
                ↓
        部分 Reduce 任务数据量过大
                ↓
        内存溢出 / 磁盘溢出 / 网络拥堵
```

## 3. 常见问题与优化

### 3.1 定位方法（四步定位法）

1. **查看任务日志**：找到长时间未完成或失败的任务
2. **分析 Key 分布**：对疑似倾斜的 Key 做频率统计
3. **检查 Shuffle 数据量**：对比各节点输入/输出数据量差异
4. **定位具体算子**：确认是 Join、GroupBy 还是其他操作导致

### 3.2 优化方案

#### 3.2.1 预处理阶段

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **过滤异常 Key** | 提前过滤掉 null 或无效 Key | 空值/默认值导致倾斜 |
| **数据打散** | 对 Key 加随机前缀后二次聚合 | GroupBy 倾斜 |
| **分桶/分区优化** | 使用更均匀的分区策略 | 数据写入阶段 |

#### 3.2.2 Join 优化

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **Map Join** | 小表广播到所有 Map 节点 | 大表 Join 小表 |
| **倾斜 Join** | 将热点 Key 单独处理 | 大表 Join 大表，部分 Key 倾斜 |
| **随机前缀 Join** | 对倾斜 Key 加随机前缀后拆分 Join | 无法使用 Map Join 时 |

#### 3.2.3 计算阶段优化

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **两阶段聚合** | 先局部聚合再全局聚合 | GroupBy 倾斜 |
| **增加 Reduce 数量** | 提高并行度 | 数据量大但 Key 分布尚可 |
| **自定义分区器** | 根据业务逻辑重写分区规则 | 默认 Hash 分区不均匀 |

#### 3.2.4 框架级调优

- **Spark**：`spark.sql.adaptive.skewJoin.enabled`（自适应倾斜 Join）
- **Hive**：`hive.groupby.skewindata=true`（GroupBy 倾斜优化）
- **Flink**：`table.optimizer.join-reorder-enabled` + 自定义分区策略

### 3.3 常见误区

- ❌ 盲目增加资源（内存/CPU）—— 不能解决分布不均问题
- ❌ 只关注一个维度 —— 倾斜可能同时发生在多个 Key 上
- ❌ 忽略数据源质量 —— 上游数据倾斜会传导到下游

## 4. 学习建议

### 4.1 知识储备

- 掌握 MapReduce / Spark / Flink 的 Shuffle 原理
- 理解 Hash 分区、Range 分区、自定义分区器的差异
- 熟悉各框架的倾斜处理参数（如 Spark AQE、Hive Skew Join）

### 4.2 实践路径

1. **从日志入手**：学会查看任务执行计划（DAG）和 Stage 耗时
2. **模拟场景**：构造 1 亿条数据，制造热点 Key 并尝试优化
3. **对比实验**：同一任务分别用 Map Join、两阶段聚合、自适应优化执行，对比耗时
4. **阅读源码**：理解 Spark `SortShuffleManager` 和 `SkewedJoin` 的实现

### 4.3 推荐资源

- **官方文档**：Spark SQL Performance Tuning、Hive Skew Join
- **经典案例**：阿里云 MaxCompute 数据倾斜调优文档
- **工具**：Spark UI（查看 Stage 数据量分布）、Hive Explain（查看执行计划）
