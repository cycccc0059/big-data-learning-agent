# 数据倾斜处理笔记

## 1. 核心概念

**数据倾斜**是指在分布式计算中，数据被分配到各个计算节点（如 Reduce Task）时，由于 key 分布不均匀，导致部分节点需要处理远超其他节点的数据量，从而拖慢整个作业的执行效率。

- **根本原因**：key 在 Reduce 阶段的分布严重不均，某些 Reduce Task 处理大量 key，而其他 Reduce Task 处理少量 key。
- **典型表现**：作业进度卡在 99% 或部分任务运行时间远长于其他任务。

## 2. 关键机制

数据倾斜的产生与 MapReduce 计算框架的工作流程密切相关：

- **Map 阶段**：读取文件，处理后以 `<key, value>` 形式输出，并对 key 进行分区和排序。
- **Shuffle 阶段**：将 Map 输出的数据按分区发送到对应的 Reduce Task。
- **Reduce 阶段**：每个 Reduce Task 处理属于自己分区的数据。

当某个分区的 key 数量远多于其他分区时，该 Reduce Task 成为瓶颈，导致数据倾斜。

## 3. 常见问题与优化

### 3.1 大量 key 为 null 值

**问题**：所有 null 值被分配到同一个 Reduce Task，造成倾斜。

**优化方案**：为 null 值赋予随机值，使其分散到不同 Reduce Task。

```sql
-- 示例：将 null 替换为随机值
SELECT CASE WHEN key IS NULL THEN CONCAT('rand_', RAND()) ELSE key END, ...
FROM table;
```

### 3.2 大量 key 不为 null 值

#### 由 GROUP BY 导致的数据倾斜

**问题**：GROUP BY 操作后，某些 key 对应的数据量极大。

**优化方案**：

1. **开启 Map 端预聚合**：减少发送到 Reduce 的数据量。
   ```sql
   SET hive.map.aggr = true;
   ```

2. **开启倾斜优化**：启动两个 Job，第一个 Job 随机分发数据到 Reduce 进行预聚合，第二个 Job 正常聚合。
   ```sql
   SET hive.groupby.skewindata = true;
   ```

#### 由 JOIN 导致的数据倾斜

**问题**：JOIN 操作中，关联键分布不均导致某些 Reduce Task 处理大量数据。

**优化方案**：

1. **大小表 JOIN（Map Join）**：将小表加载到内存，在 Map 阶段完成 JOIN，避免 Reduce 阶段。
   ```sql
   -- 自动开启 Map Join
   SET hive.auto.convert.join = true;
   SET hive.mapjoin.smalltable.filesize = 25000000; -- 小表大小阈值
   ```

2. **大表与大表 JOIN（Skew Join）**：对倾斜的 key 单独处理。
   - 正常数据走普通 Reduce JOIN。
   - 倾斜 key 的数据先写入 HDFS，再启动一轮 Map JOIN 单独处理。
   ```sql
   SET hive.optimize.skewjoin = true;
   SET hive.skewjoin.key = 100000; -- 倾斜 key 的数量阈值
   ```

## 4. 学习建议

1. **理解底层原理**：掌握 MapReduce 的 Shuffle 和分区机制，是定位数据倾斜问题的基础。
2. **学会定位倾斜**：通过查看作业日志、任务运行时间、数据量统计，判断是否存在倾斜及倾斜的 key。
3. **优先使用参数优化**：对于 GROUP BY 和 JOIN 场景，先尝试开启 Hive 内置的倾斜优化参数。
4. **结合业务调整**：对于特定业务逻辑（如 null 值、热点 key），手动改写 SQL 或调整数据分布往往更有效。
5. **关注数据预处理**：在 ETL 阶段对数据进行均匀分桶或加盐处理，可以从源头减少倾斜风险。
