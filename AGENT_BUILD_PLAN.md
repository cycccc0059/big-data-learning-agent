# 大数据开发学习 Agent 构建与迭代计划

这份文档用于记录当前 Agent 的建设路线。它不是项目使用说明，项目怎么运行请看 `README.md`；这里主要回答三个问题：

1. 这个 Agent 最终要变成什么。
2. 从现在的 Demo 到可用 Agent，中间要补哪些能力。
3. 后续我们在聊天框里如何一小步一小步实现。

## 1. 当前 Demo 状态

当前项目已经完成 `v0.1` 的最小可运行版本，核心目标是先跑通 Agent 的基本闭环。

已有能力：

- 命令行入口：通过 `python3 agent.py` 启动对话。
- Agent 角色设定：定位为大数据开发方向的学习与工作问题助手。
- 本地记忆：最近对话会保存到 `data/memory.json`。
- Markdown 笔记读取：会读取 `notes/` 目录下的本地知识笔记。
- 可选 LLM 接入：配置 `OPENAI_API_KEY` 后，可以调用 OpenAI 兼容接口。
- 本地规则模式：没有 API Key 时，也能用规则模板演示基本流程。

当前边界：

- 还没有真正的知识库检索，只是简单读取 Markdown 内容。
- 还没有问题分类能力。
- 还没有向量数据库。
- 还没有工具调用，例如 SQL 分析、日志解析、代码读取。
- 还没有 Web UI。
- 还没有学习进度追踪和复盘机制。

## 2. 最终目标

这个 Agent 的目标不是做一个泛泛的聊天机器人，而是做一个面向大数据开发方向的个人学习与问答助手。

它应该逐步具备这些能力：

- 学习规划：根据当前基础，生成 Spark、Flink、Hive、Kafka、数仓等学习路线。
- 概念问答：解释大数据组件的核心概念、机制和应用场景。
- 工作排查：针对 SQL 慢、Spark 任务失败、Flink 延迟高、Kafka lag、数据异常等问题给排查路径。
- 项目实操：帮助设计离线数仓、实时链路、数据质量校验、调度流程等 Demo 项目。
- 面试准备：整理高频面试题、回答框架、实战案例和追问方向。
- 知识沉淀：把高质量回答、学习记录、排查经验整理成本地知识库。
- 个性化记忆：逐步了解用户当前基础、学习目标、薄弱点和正在推进的项目。

一个理想的提问与回答流程是：

```text
用户问题
-> 判断问题类型
-> 检索本地知识库
-> 结合最近对话和个人学习档案
-> 生成结构化回答
-> 给出下一步行动
-> 必要时沉淀为笔记
```

## 3. 核心模块清单

### 3.1 模型层

负责调用大模型并生成回答。

初期建议：

- 先使用 OpenAI 兼容接口，保持实现简单。
- 保留 `OPENAI_BASE_URL`，后续可以接 DeepSeek、Qwen 或其他兼容服务。
- 暂时不引入复杂 Agent 框架，先理解核心流程。

后续可选：

- 接入本地模型，例如 Ollama。
- 支持不同任务使用不同模型。
- 增加模型调用失败时的降级策略。

### 3.2 知识库层

负责保存学习资料、案例、个人记录。

知识库建议使用 Markdown 起步，原因是：

- 容易手写和维护。
- 适合版本管理。
- 方便后续切块和向量化。

建议目录结构：

```text
knowledge/
├── profile/       # 个人目标、当前水平、薄弱点
├── roadmap/       # 学习路线
├── components/    # Hadoop、Hive、Spark、Flink、Kafka 等组件知识
├── cases/         # 工作问题和排查案例
├── interview/     # 面试题与回答框架
└── projects/      # 实战项目设计与复盘
```

### 3.3 检索层

负责从知识库中找到与问题相关的资料。

阶段设计：

- `v0.2`：先做文件级或关键词级检索。
- `v0.3`：加入文档切块、向量化和向量数据库。
- 后续：回答时标注参考来源，减少纯模型幻觉。

推荐技术：

- Markdown 文件作为原始资料。
- Chroma 或 FAISS 作为初版向量库。
- 后续如果需要服务化，再考虑 Qdrant、Milvus 或 pgvector。

### 3.4 任务规划层

负责把模糊问题拆成可执行步骤。

典型场景：

- “我想学习 Spark” -> 拆成周期、主题、练习和检查标准。
- “Flink 延迟高怎么办” -> 拆成现象确认、指标查看、原因定位、解决方案。
- “我要做一个数仓项目” -> 拆成数据源、分层建模、指标体系、调度和质量校验。

### 3.5 工具层

负责让 Agent 不只会回答，还能执行辅助动作。

可逐步增加：

- 读取本地文件。
- 整理 Markdown 笔记。
- 分析 SQL 文本。
- 解释错误日志。
- 生成学习计划。
- 生成练习题。
- 生成项目目录和示例代码。

### 3.6 记忆层

分为短期记忆和长期记忆。

短期记忆：

- 当前对话上下文。
- 最近几轮问题和回答。

长期记忆：

- 用户学习目标。
- 当前技术基础。
- 已学习内容。
- 薄弱点。
- 正在做的项目。
- 常见工作问题。

初期可以继续使用 JSON；当数据变多后，再考虑 SQLite。

### 3.7 评估层

用于判断 Agent 回答是否真的有用。

每次回答可以从这些角度检查：

- 是否回答了用户问题。
- 是否给出可执行步骤。
- 是否结合了知识库内容。
- 是否说明了假设和限制。
- 是否给出下一步行动。
- 是否适合用户当前阶段。

## 4. 知识库构建方案

即使当前没有本地资料，也可以从小规模知识库开始。不要一开始追求资料齐全，先建立结构，再逐步填充。

### 4.1 第一类：基础知识库

用于覆盖大数据开发核心技术栈。

建议先建这些主题：

- 大数据整体学习路线。
- Hadoop、HDFS、YARN。
- Hive 与离线数仓。
- Spark 与 Spark SQL。
- Flink 与实时计算。
- Kafka 与消息队列。
- 调度系统。
- 数据治理和数据质量。
- 湖仓技术。

每篇笔记建议包含：

```text
核心概念
常见使用场景
重要机制
常见问题
实操练习
检查标准
```

### 4.2 第二类：问题案例库

这是后续最有价值的部分，因为它会让 Agent 更贴近真实工作。

建议先沉淀这些案例：

- Spark SQL 慢查询排查。
- Spark 任务 OOM 排查。
- Flink checkpoint 失败排查。
- Flink 任务延迟升高排查。
- Kafka lag 过高排查。
- Hive 分区数据异常排查。
- 数据质量异常排查。

每个案例建议使用固定模板：

```text
现象
背景
可能原因
排查步骤
验证方式
解决方案
复盘总结
```

### 4.3 第三类：个人学习记录

用于让 Agent 越来越了解用户。

建议记录：

- 当前学习目标。
- 当前技术水平。
- 每周学习计划。
- 每次学习复盘。
- 薄弱点列表。
- 项目想法。
- 面试准备记录。

这些内容不需要一开始很完整，可以在每次对话后慢慢补。

## 5. 迭代路线

### v0.1：最小可运行 Demo

目标：

- 跑通命令行对话、基础 Prompt、本地记忆和 Markdown 读取。

已完成：

- CLI 对话入口。
- 大数据开发 Agent 角色设定。
- 本地规则模式。
- 可选 LLM 接入。
- 简单本地记忆。
- 简单读取 `notes/`。

验收标准：

- 执行 `python3 agent.py` 能启动。
- 输入问题后能返回结构化文本。
- 对话能写入 `data/memory.json`。

### v0.2：真实学习问答基础版

目标：

- 让 Agent 更像一个大数据学习助手，而不是通用模板回答器。

要做：

- 建立标准知识库目录。
- 增加 `.env.example`，说明模型配置方式。
- 增加问题分类能力。
- 增加学习类回答模板。
- 增加排查类回答模板。
- 整理第一批大数据知识库模板。

建议问题类型：

- 学习规划。
- 概念解释。
- 工作排查。
- SQL 优化。
- 项目设计。
- 面试准备。
- 笔记整理。

验收标准：

- 输入“我想学习 Spark SQL 优化”，能返回学习路径、练习任务和检查标准。
- 输入“Flink 延迟高怎么排查”，能返回排查步骤、验证方式和解决方案。
- 输出结构明显区分学习类问题和排查类问题。

### v0.3：知识库检索版

目标：

- 让 Agent 回答前先检索知识库，减少只靠模型记忆。

要做：

- 将 Markdown 文档切块。
- 为每个文档片段建立索引。
- 接入 Chroma 或 FAISS。
- 根据用户问题检索相关片段。
- 回答时附带参考来源。

验收标准：

- 问 Spark 相关问题时，能优先检索 Spark 笔记。
- 问 Flink 相关问题时，能优先检索 Flink 笔记。
- 回答中能说明参考了哪些本地资料。

### v0.4：学习计划与复盘版

目标：

- 让 Agent 能陪伴持续学习，而不是一次性问答。

要做：

- 生成 7 天或 4 周学习计划。
- 保存学习目标和当前水平。
- 支持每日学习任务。
- 支持学习复盘。
- 自动记录薄弱点。

验收标准：

- 可以生成一份 Spark 或 Flink 的 4 周学习计划。
- 可以根据复盘内容更新薄弱点。
- 可以给出下一次学习建议。

### v0.5：工作问题处理版

目标：

- 让 Agent 能辅助处理真实大数据开发问题。

要做：

- SQL 优化分析模板。
- Spark 错误日志解释模板。
- Flink 延迟与 checkpoint 排查模板。
- Kafka lag 排查模板。
- 数仓建模建议模板。
- 数据质量排查模板。

验收标准：

- 用户贴一段 SQL，Agent 能给出优化方向。
- 用户描述一个任务失败现象，Agent 能给出排查顺序。
- 用户描述一个指标异常，Agent 能给出数据链路排查方案。

### v0.6：界面与工具化版本

目标：

- 让 Agent 从命令行工具变成更顺手的日常工具。

可选方向：

- Streamlit Web UI。
- FastAPI 服务。
- 本地桌面应用。
- 飞书或企业微信机器人。

推荐优先做：

- Streamlit Web UI，因为开发成本低，适合学习型 Agent。

验收标准：

- 可以在浏览器中输入问题。
- 可以查看历史记录。
- 可以选择知识库或问题类型。
- 可以把好回答保存为 Markdown 笔记。

## 6. 每一步聊天协作方式

后续我们在聊天框里推进时，建议按小版本、小闭环进行。

每次迭代遵循这个流程：

```text
确定本次目标
-> 明确成功标准
-> 修改或新增最少必要文件
-> 运行验证
-> 总结本次结果
-> 记录下一步
```

协作原则：

- 一次只实现一个核心能力。
- 每个能力都要能运行验证。
- 不为了复杂而复杂，先保证能用，再逐步增强。
- 新增功能前，先确认它解决什么具体问题。
- 每次迭代后，都把经验沉淀进文档或知识库。

## 7. 下一步建议

下一步从 `v0.2` 开始，不急着接向量数据库。

建议第一轮实现内容：

1. 建立标准知识库目录 `knowledge/`。
2. 新增 `.env.example`。
3. 增加问题分类逻辑。
4. 增加学习类和排查类回答模板。
5. 新增第一批知识库模板文件。

完成后，Agent 就会从“能跑的 Demo”变成“有明确大数据学习方向的基础问答工具”。
